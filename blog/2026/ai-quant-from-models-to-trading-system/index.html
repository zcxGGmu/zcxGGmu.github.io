AI 量化：从模型到交易系统
AI 量化的核心不是把行情和新闻丢给大模型让它喊买卖，而是把数据、特征、模型、风险、执行、回测、校准和审计做成一个闭环。模型可以提供优势，但真正长期复利的是工程系统。
2026-06-07  ▣ 投资 AI  ◇ AI 量化 量化交易 交易系统 风险管理 强化学习  ◷ 13 min

这篇文章整合自两篇英文原文：AI Quantitative Trading: From Models to Quant Funds 与 Dnalyaw: Engineering an AI Quant Trading System from Scratch。本文不是逐字翻译，而是保留核心判断与技术脉络后形成的中文融合版：先解释 AI 为什么天然适合一部分量化问题，再拆解一个从研究到实盘执行的量化交易系统应该如何设计。

AI 量化最容易被误解的地方，是把它想象成一个“会看 K 线、会读新闻、会给买卖建议的大模型”。这种想象很直观，也很危险。真正可用的 AI 量化不是让语言模型直接决策仓位，而是把市场数据、非结构化信息、统计因子、机器学习模型、强化学习策略、风险约束、订单管理、成交校准和可观测性串成一个闭环。模型只是其中一层，系统才是壁垒。

本文只讨论方法和工程，不构成任何投资建议。

AI 为什么适合量化

大语言模型在很多真实世界任务里仍然受限。文本、图像、代码等输出本质上是在训练数据分布上的概率近似；一旦任务要求持续与现实世界交互、承担明确后果、处理物理约束或复杂业务责任，模型能力就会遇到边界。企业应用、机器人、自动驾驶之所以推进艰难，往往不是因为模型不能生成答案，而是因为现实环境不会按训练分布配合。

金融市场有一类完全不同的问题形态。它天然产生结构化数值数据，收益和亏损提供清晰反馈，历史数据可以被反复回测，决策目标可以形式化为收益、风险、回撤、换手、滑点、容量等指标。对 AI 来说，这类问题不需要理解现实世界的全部因果，只需要在给定状态下选择更优动作，并在长期样本中验证结果。

这并不意味着“AI 可以轻松交易”。更准确的说法是：量化交易比许多现实交互问题更适合用机器学习形式化，但真正困难的部分从来不只是模型训练，而是数据、执行、风险和资金管理。

传统量化交易也没有外界想象得那么神秘。很多策略建立在已经存在几十年的技术指标、统计因子和固定公式之上，例如动量、价值、质量、低波动、均值回归、配对交易、协整关系、RSI、MACD 等。策略开发通常是选择指标、组合参数、设定信号、做历史回测，再根据表现迭代。

从数学上看，量化策略可以理解为一个复杂函数：输入是价格、成交量、基本面、情绪、宏观、仓位和市场状态，输出是买入、卖出、空仓或目标权重。传统量化更多依赖人为定义的特征和参数搜索；AI 量化则希望让模型在更大的参数空间里学习市场状态与动作之间的关系。

这里必须区分两件事。把 OHLCV、新闻和财报摘要丢给 GPT，让它直接输出“买入”或“卖出”，并不是严肃量化。大语言模型没有天然接受过市场专用目标函数训练，也没有内置风险约束、交易成本、订单执行和资金管理。它可以帮助处理文本、抽取信息、生成假设，但不能直接替代交易系统。

量化基金的神秘感来自哪里

量化基金看起来神秘，主要不是因为每个策略都拥有无法理解的高深数学，而是因为存在强烈的信息不对称和运营门槛。真正开一个基金，需要资本、牌照、交易通道、行业经验、券商关系、风控体系、基础设施，以及能在市场里长期存活的组织能力。多数个人投资者没有机会进入这个完整环境，因此容易把量化收益误认为纯粹来自智力或公式。

顶级量化机构的优势通常来自几类复合壁垒。

高频交易依赖速度。服务器是否靠近交易所撮合引擎，网络延迟能否压到微秒级，代码路径是否足够短，硬件和协议是否优化到极致，这些往往比模型复杂度更重要。高频做市还可能通过返佣、报价优势和订单簿微结构获得收益。

波动率环境会放大机会。市场越剧烈，定价误差、流动性缺口和短期行为偏差越容易出现，优秀系统越能捕捉差价。但波动同样放大风险，只有执行和风控足够硬，才能把机会变成收益。

趋势判断也会造成巨大差异。长牛市场里，对基本面和市场方向的判断如果持续正确，收益会非常可观。2019 到 2024 年的美股环境就让不少趋势型、科技成长型和系统性策略获得了丰厚回报。

先发优势也很重要。早期量化机构面对的是更慢、更不技术化的对手；随着市场参与者技术水平提高，公开因子的边际收益会下降，拥挤交易和 alpha 衰减会越来越快。

因此，量化基金的护城河不是某个单独公式，而是数据规模、反应速度、交易成本、风险纪律、资金容量、人才密度和执行体系共同构成的系统能力。

AI 量化的正确问题形式

AI 量化最有价值的表达方式，是把交易问题形式化为状态、动作和奖励。

状态可以包括行情数据、技术指标、波动率、市场 regime、组合杠杆、回撤、历史 Sharpe、个股特征、成分股关系、新闻情绪、财报文本、管理层变化、内部人交易、宏观变量等。

动作不应该是“下一笔订单买多少股”，而应该是目标组合权重。模型决定希望持有什么样的组合，执行系统再把当前持仓转换为目标持仓，处理拆单、撮合、部分成交、撤单、结算和滑点。

奖励也不能只看收益。更合理的奖励函数需要综合 PnL、风险调整收益、最大回撤、换手、交易成本、杠杆约束、集中度约束、波动约束和违约惩罚。否则模型很容易学出短期高收益但长期必爆仓的策略。

强化学习适合这个框架，因为它本来就是在状态中选择动作，并通过奖励优化长期结果。相比几十个手工参数，拥有大量可学习参数的模型理论上能表示更复杂的市场分布。但模型容量本身不是胜利，训练数据质量、样本周期、市场 regime 覆盖、交易成本建模和风险约束同样决定结果。

很多公开论文喜欢在 2020 到 2024 年的标普 500 样本上展示回测收益，这类样本很容易被长期牛市“抬着走”。真正严肃的系统必须跨越牛市、熊市、震荡、市值风格切换、利率环境变化、流动性冲击和政策突发事件。一个只在单一阶段表现优异的模型，很可能只是学会了那段市场的方向。

真正的壁垒：不是模型，而是闭环

严肃 AI 量化至少需要解决六个问题。

第一，数据规模和质量。金融数据噪声极高，特征越多不等于信号越强。系统必须处理复权、幸存者偏差、前视偏差、时间戳对齐、公告时点、成交可得性、停牌、异常报价和供应商差异。

第二，人类经验如何进入系统。交易员经验不应该只存在于口头判断里，可以转化为策略先验、风险边界、特征工程、样本筛选、异常场景标签和行为约束。

第三，对冲是基本能力。市场不是静态分类问题，正确方向也可能因为仓位、相关性、杠杆和波动暴露而亏损。模型必须管理组合风险，而不是只追逐单个信号。

第四，资金管理本身就是模型。一个普通策略配合优秀仓位控制，可能长期跑赢一个信号很强但仓位失控的策略。量化系统里，position sizing 与 alpha 同等重要。

第五，不应该一个模型吃遍所有市场。资产类别、交易周期、波动结构、参与者组成、流动性和监管环境差异巨大，不同策略、不同市场和不同 regime 往往需要不同模型。

第六，过滤噪声是核心竞争力。金融市场数据充沛但极脏，任何看起来显著的关系都可能只是偶然、拥挤、过拟合或回测污染。真正困难的不是产生因子，而是判断因子是否仍然有未来收益。

从零构建交易系统：为什么要分层

一个面向实盘的 AI 量化系统不能用“Python 从头写到尾”的方式处理所有问题。研究阶段可以追求灵活，但订单执行和风险检查必须尊重延迟层级。毫秒、微秒甚至纳秒级路径里，垃圾回收、动态类型、网络抖动和不确定的中间件都可能成为风险。

Dnalyaw 这类系统的关键思想，是用不同语言承接不同延迟层级。

Rust 负责热路径。风险引擎和执行核心运行在 Rust 中，每一笔订单都必须经过交易前风险检查。Rust 的价值不是“更酷”，而是无垃圾回收暂停、内存安全、可预测延迟和零分配热路径。风险检查不能因为某个运行时暂停而排队，更不能被策略绕过。

Go 负责温路径。订单管理系统、策略编排、API 层和实时监控适合放在 Go 中。goroutine 和 channel 很适合市场数据分发、策略 fan-out、目标组合收集、订单生命周期管理和状态机处理。Go 足够快，也比 Rust 更适合快速迭代业务编排。

Python 负责冷路径。研究、回测、机器学习训练、强化学习、LLM 集成和特征工程仍然应该放在 Python 中。Python 的生态和开发效率在研究侧非常重要，但 Python 策略不应该直接发订单。更合理的边界是：Python 输出目标组合权重，经由 gRPC 或类似协议进入 Go 执行层，再由 Rust 风险引擎决定能否执行。

这种分层背后的原则很简单：越接近真实资金和订单，越必须确定、可控、低延迟、不可绕过；越接近研究探索，越可以灵活、异步、快速试错。

风险引擎必须拥有否决权

交易系统里最重要的设计，不是模型多聪明，而是风险引擎拥有绝对否决权。风险管理不能只是策略代码上的一个 decorator，也不能是“最好努力”的中间件。只要真实资金在场，风险系统就必须在架构上不可绕过。

每笔订单可以被风险引擎判定为四种结果。

APPROVE 表示按请求执行。

REJECT 表示完全阻止，例如触发限额、市场关闭、集中度过高或订单不合法。

REDUCE 表示缩小订单规模。这个判断很关键，因为策略方向可能是对的，但仓位过大。缩小订单既保留信号，也维护风险边界。

FLATTEN 表示进入紧急模式，关闭所有仓位，并切换到只减仓状态。

有些风险限制应该写成不可变常量，而不是运行时配置。比如单一标的最大名义敞口、行业敞口上限、日内回撤熔断、单笔交易最大风险等。原因并不复杂：当仓位正在流血时，人最容易说服自己“这次例外”。可配置的限制会被重新配置，写死的限制才会在压力下保持纪律。

仓位管理可以同时使用进攻与防守两套模型。Kelly Criterion 用于估计长期复利意义下的最大下注比例，实践中常用 Half-Kelly 降低估计误差和肥尾风险；Van Tharp 的 R-Multiple 框架则强调单笔亏损不得超过资本的一定比例。最终仓位应该取所有约束中最保守的结果。机会决定上限，生存决定底线。

执行系统决定收益能否落地

回测里赚钱，不代表实盘里能赚钱。大多数策略从研究走向实盘时会失败，核心原因是成交假设不真实。纸面上 Sharpe 2.0 的策略，上线后可能只有 0.3，因为滑点、冲击成本、限价单不成交、借券成本、撮合延迟、交易所状态和资金容量都会侵蚀收益。

因此，执行系统不能是事后补上的模块，而必须从第一天就是核心系统。

外部数据源、券商和交易所应该通过统一 provider abstraction 接入。不同供应商可能使用二进制协议、TCP stream、REST、protobuf 或私有接口，但下游不应该感知这些差异。系统内部需要标准化符号、报价、订单、成交、账户、仓位和市场状态。

市场数据进入策略层之前，还必须经过质量门禁。异常 bid-ask spread、过期时间戳、统计异常价格跳变、断流和坏 tick 都要被过滤。坏报价如果进入策略层，可能触发一连串错误订单；质量门禁的成本很低，但能避免大量灾难。

多市场执行还会引入区域部署问题。美股基础设施需要靠近 NYSE/NASDAQ，港股基础设施需要靠近 HKEX。如果同一区域能同时接入不同 venue path，还可以持续比较成交质量，这相当于在真实市场中做执行 A/B 测试。

从回测到实盘：校准比漂亮曲线更重要

严肃系统需要两层回测。

第一层是快速 Python 向量化回测，用于在几秒内验证信号逻辑。这一层服务研究迭代，关注速度和便利。

第二层是集成 OMS 的 Go 回测，用模拟交易所跑真实订单状态机。它能暴露生产环境才会出现的问题，例如部分成交、订单拒绝、状态幂等、结算时间、撤改单逻辑和券商状态对账。

真实成交建模至少要纳入四类因素：买卖价差滑点、基于成交量的市场冲击、限价单成交概率，以及根据真实延迟校准的执行等待时间。价格不会因为策略需要成交而成交，订单自身也会推动价格朝不利方向移动。

策略正式分配资金之前，应该先做影子交易。策略在实时市场数据上并行运行，但不真正下单；系统持续比较模型预期 PnL 与真实可执行结果之间的偏差，观察每日收益、Sharpe、成交率和平均滑点等指标。如果漂移过大，就说明回测假设与现实不匹配。

资金部署也应该分阶段：验证、校准、小额分配、完整分配。策略晋级不是因为历史曲线好看，而是因为实时执行结果在容忍范围内匹配模型预期。

LLM 是特征源，不是护城河

AI 交易叙事里最容易膨胀的部分是 LLM。把行情和新闻交给大模型，让它直接输出买卖信号，本质上是高度拥挤、容易复制、难以验证的做法。任何存在于公共 API 端点上的语言模型能力，都很难成为持久 alpha。

LLM 真正有价值的位置，是把非结构化信息转化为可进入特征库的数值特征。10-K 年报可以变成诉讼风险评分，财报电话会可以变成语调变化向量，内幕交易披露可以变成情绪数值，管理层变动可以变成类别特征，新闻可以变成事件冲击因子。这些特征再与价格、成交量、基本面、宏观、技术指标和组合状态放在同一个特征矩阵里。

换句话说，LLM 不应该生成订单、决定仓位、修改风控参数、覆盖止损或触碰执行路径。它只是众多特征生产者之一。真正决定组合持有什么的，是确定性的量化管线与风险系统。

这也解释了为什么护城河不在单个模型，而在垂直整合管线。一个 LLM 是商品；一个能把几十种异构来源、上百个特征、回测、校准、执行、风控和审计汇聚到同一闭环里的系统，才可能形成复利优势。

强化学习负责动作，执行系统负责落地

强化学习在交易中的定位也需要清晰。研究层产生信号，包括 alpha 因子、技术指标、LLM 情绪分数、宏观状态、组合状态和风险指标；RL agent 的任务不是直接下订单，而是在给定状态下输出目标组合权重。

这种拆分让强化学习专注于决策：在当前市场 regime、持仓、回撤、杠杆、波动率、个股特征和配对关系下，组合应该长什么样。执行系统则负责把当前持仓移动到目标持仓，处理路由、拆单、部分成交和结算。

模型上可以探索 PPO 与 SAC。PPO 在非平稳市场中更稳定，SAC 在数据昂贵时更具样本效率。为了适应 regime 变化，滚动重训也很重要，例如按固定周期更新训练窗口，并监控模型漂移。

更进一步的方向，是把神经学习与符号约束结合起来。纯神经模型可以从数据中学模式，但很难保证逻辑约束；纯规则系统可解释但脆弱。神经符号方法可以把硬性交易约束、市场结构关系和可学习嵌入放入同一计算框架：训练阶段用连续模式学习关系，实盘关键约束切换为布尔模式，保证学到的策略不能违反风险边界。

这类方法的意义不是让系统显得复杂，而是避免“黑箱模型学到了好看的回测，同时偷偷违反了交易纪律”。在金融系统里，可解释、可组合、可约束往往比单纯更大的模型更重要。

Agent 可以挖因子，但不能直接交易

Agent 在量化研究里有一个很自然的位置：生成假设。它可以提出类似“研发支出占收入比例持续上升的公司，未来六个月是否更容易跑赢”这样的因子想法；回测验证器再用历史数据测试；衰减检查器再判断它是否过于接近已有拥挤因子。

这种流程里，Agent 是研究助理，不是交易员。每个候选因子仍然要通过回测、校准、影子交易、风险检查和分阶段资金分配。高效市场中，一个所有人都能同时发现的因子很快会衰减，因此系统必须显式检查新因子的独特性、拥挤度和持续性。

这比让 Agent 直接“读新闻下单”稳得多。前者把 AI 放在可验证流程中，后者把资金交给不可审计的即时判断。

可观测性必须内建

实盘系统没有可观测性，就等于没有系统。每个组件都应该在实现时同时内建日志、指标和审计，而不是上线后再补。

结构化日志需要 correlation ID，能把一笔订单从信号生成、风险检查、OMS、venue submission 到 fill receipt 全链路串起来。

指标系统需要延迟直方图、订单状态计数、venue 维度、symbol 维度、敞口指标、P&L、成交质量和错误率。OpenTelemetry、Prometheus 或类似方案的价值，是让延迟和风险从感觉变成可度量事实。

审计轨迹需要不可变记录。每次状态变更都要包含时间戳、actor、完整 payload，以及风险决策时的指标快照。否则出现问题后，只能猜测系统当时为什么那样做。

实时运维界面不一定要复杂。很多专业交易环境偏好 Grafana 加终端工具，因为它们快、稳定、故障面小。TUI 可以展示仓位、实时报价、订单簿、风险面板、告警和 watchlist；关键不是界面炫，而是压力下可靠。

五个工程结论

第一，先建执行，再做策略。没有可靠执行，就无法真实测试策略。很多量化项目从模型开始，最后再补订单管理和风控，这是反过来的。

第二，回测永远会撒谎。不是因为回测没有价值，而是因为它无法完整模拟滑点、借券、市场冲击、容量、延迟和交易所细节。影子交易和实盘校准是解药。

第三，对 alpha 要诚实。成熟市场里，很多统计套利策略扣除成本后的年化空间并没有想象中夸张。Sharpe 超过 1 已经不错，超过 2 需要认真检查假设，回测显示 50% 年化以上往往意味着哪里出了问题。

第四，优势不只是模型，而是执行。成交好坏、滑点控制、订单路由和交易成本会在成千上万笔交易中复利。长期看，一点点执行差异会吞掉大量理论 alpha。

第五，风险工程比 alpha 更重要。一个普通策略配合坚固风控可以活到下一天；一个天才策略配合糟糕风控，迟早会在极端场景里出局。交易系统首先要解决生存，然后才谈收益。

结语

AI 量化真正值得关注的，不是“模型会不会预测明天涨跌”，而是它能否进入一个可验证、可回放、可审计、可校准的工程闭环。数据提供状态，模型产生信号，LLM 转换非结构化信息，RL 输出目标组合，风险引擎拥有否决权，执行系统负责落地，回测与影子交易持续校准，可观测性记录每一次状态变化。

当这些环节被放进同一条从研究到实盘的管线里，AI 才不是噱头，而是杠杆。单个模型会衰减，单个因子会拥挤，单次回测会失真；但一个能持续发现、验证、执行、纠错和审计的系统，才有机会把短期聪明变成长期复利。
